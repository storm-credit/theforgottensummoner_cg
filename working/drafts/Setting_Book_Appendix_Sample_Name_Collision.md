# Setting Book Appendix Sample: Name Collision Control

## Purpose

이 appendix 샘플은 비슷한 이름이 여러 대륙, 세력, 역할, 혹은 유물 층에서 겹칠 때
설정집이 어떻게 병합 사고를 막아야 하는지 보여준다.

본문은 깨끗한 이름을 써야 하지만,
부록은 어떤 이름이 아직 분리 보존 대상인지 선명하게 남겨야 한다.

## Collision Rule

이름이 같다고 같은 존재는 아니다.

다음 경우에는 즉시 병합하지 않는다.

- 대륙 앵커가 다를 때
- 세력 앵커가 다를 때
- 성, 이명, 직함이 다를 때
- 인물명과 유물명, 마법명, 개념명이 겹칠 때
- 구버전 표기와 별도 인물 가능성이 모두 열려 있을 때

## Body-Safe Naming Rule

본문에서는 아래 순서를 따른다.

1. 현재 가장 안전한 display name을 쓴다.
2. 같은 이름군 안의 다른 후보가 있으면 필요한 경우 짧은 변형 식별자를 붙인다.
3. 본문 안에서 병합 서사를 새로 만들지 않는다.
4. 병합이 안 끝난 이름은 appendix에서만 상세 관리한다.

예:

- `Sylvia` 단독 표기가 위험하면 `Oceanic Sylvia`처럼 임시 식별자를 쓴다.
- `Aegis`는 필요 시 인물/유물/방벽 개념 중 어느 층인지 함께 밝힌다.
- `Isador` 단독 표기 대신 `Isador Tempest` 혹은 `Isadore Solea`처럼 가문/탑 앵커를 붙인다.

## Misread Risk Notes

이름 충돌이 위험한 이유는 단순한 오탈자 문제가 아니기 때문이다.

- 잘못 병합하면 서로 다른 대륙의 서사가 한 인물로 압축된다.
- 인물과 유물을 섞으면 아이템 canon과 인물 canon이 동시에 오염된다.
- 구버전 표기를 성급히 정본화하면 이후 원본 대조가 어려워진다.
- 독자용 본문에서 하나로 써 버리면 나중에 다시 분리하기가 훨씬 힘들어진다.

## Body Conversion Example

Technical appendix language:

```text
Cluster: Sylvia
Handling: separate candidates
Risk: continent and role collision
```

Reader-facing body language:

```text
실비아라는 이름은 이 세계에서 하나의 얼굴만 가리키지 않는다. 어떤 실비아는 기록과 시약의 탑에 남아 있고, 어떤 실비아는 바다의 흐름 속에서 전설처럼 떠돈다. 그래서 설정집은 같은 이름을 발견해도 먼저 누구의 이야기인지부터 확인한다.
```

## Promotion Gate

하나의 이름군을 단일 본문 엔트리로 승격하려면 아래가 필요하다.

1. 동일한 대륙/세력/가문 앵커가 반복 확인된다.
2. 성, 이명, 역할 신호가 충돌하지 않는다.
3. 인물/유물/개념 층이 분리된다.
4. 구버전 표기인지 별도 인물인지 source check가 끝난다.

## Conductor Decision

이 appendix 샘플의 목적은 이름을 늘리는 것이 아니라
이름을 잘못 줄이지 않는 것이다.

좋은 설정집은 멋진 이름을 많이 가지는 책이 아니라,
각 이름이 정확히 누구와 무엇을 가리키는지 끝까지 조심하는 책이어야 한다.
